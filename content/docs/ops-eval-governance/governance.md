## NIST AI RMF 기반 에이전트 거버넌스

NIST AI Risk Management Framework (AI RMF)는 에이전트 거버넌스의 실무 기반으로 가장 널리 인정받습니다.

**4가지 핵심 기능**

| 기능 | 설명 | 에이전트 적용 |
|------|------|-------------|
| **GOVERN** | AI 리스크 관리 문화·정책 수립 | 에이전트 정책, 책임 체계 |
| **MAP** | AI 리스크 식별·분류 | 에이전트 공격면 매핑 |
| **MEASURE** | 리스크 측정·모니터링 | Eval, 관측성 |
| **MANAGE** | 리스크 대응·개선 | 인시던트 대응, 업데이트 |

## 조직 정책 설계

에이전트 도입 시 정의해야 할 핵심 정책:

### 에이전트 분류 정책

| 등급 | 기준 | 승인 절차 |
|------|------|---------|
| **저위험** | 읽기 전용, 내부 사용 | 팀장 승인 |
| **중위험** | 제한된 쓰기, 내부 사용 | 부서장 + 보안팀 |
| **고위험** | 외부 노출, 재무/법무 영향 | CISO + 경영진 |

### 최소 권한 원칙 (Least Privilege)

```
❌ 나쁜 예:
에이전트에게 전체 DB 읽기/쓰기 권한 부여

✅ 좋은 예:
- 고객 주문 조회: SELECT on orders WHERE customer_id = $user_id
- 환불 처리: EXECUTE on stored_proc_process_refund
- 개인정보: 마스킹된 필드만 접근
```

## 에이전트 신원과 감사

### KYA (Know Your Agent)

2026년 현재 에이전트 신원 인증은 미해결 과제입니다.
현재 실무에서 사용하는 방법:

1. **API 키 기반 인증**: 에이전트별 고유 API 키
2. **행동 감사 로그**: 에이전트가 실행한 모든 행동 기록
3. **세션 ID 추적**: 요청부터 완료까지 추적 가능한 ID

## 벤더·모델 리스크 관리

| 리스크 | 대응 전략 |
|--------|---------|
| 모델 API 가용성 | 멀티 모델 폴백 설계 |
| 모델 성능 변화 | 버전 고정 + 변경 시 eval 재실행 |
| 데이터 프라이버시 | 온프레미스 또는 프라이빗 API 검토 |
| 비용 급증 | 사용량 알림 + 하드 제한 |
| 벤더 종속성 | 프로토콜 표준(MCP/A2A) 기반 설계 |
