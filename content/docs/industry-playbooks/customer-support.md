## 문제 정의

고객지원팀이 처리하는 티켓의 60-70%는 반복적인 유형입니다. 동시에 고객은 빠른 응답을 기대하고, 상담원은 반복 업무에 지쳐있습니다.

**자동화 목표**: 1차 응답 시간 단축 + 상담원이 복잡한 케이스에 집중

## 추천 아키텍처

```
고객 문의 입력
      ↓
[분류 에이전트]
  ├── FAQ/단순 문의 → [RAG 자동 응답]
  ├── 주문/환불 문의 → [주문 처리 에이전트]
  └── 복잡/불만 → [에스컬레이션 → 상담원]
```

**추천 패턴**: 라우팅 + 단일 에이전트 (유형별)

## 핵심 도구 목록

| 도구 | 기능 | 권한 수준 |
|------|------|---------|
| search_faq | FAQ 검색 | 읽기 |
| get_order | 주문 조회 | 읽기 |
| get_customer | 고객 정보 조회 | 읽기 (마스킹) |
| update_ticket | 티켓 상태 업데이트 | 쓰기 |
| process_refund | 환불 처리 | 쓰기 + 승인 |
| escalate_to_agent | 상담원 배정 | 쓰기 |

## 위험 체크리스트

- [ ] PII (개인정보) 로그에 기록되지 않도록 마스킹
- [ ] 환불 처리는 반드시 금액 한도 설정
- [ ] 감정이 높은 고객은 에스컬레이션 트리거
- [ ] 에이전트 응답에 "AI가 답변한다"는 사실 명시
- [ ] 고객이 상담원 연결 요청 시 즉시 에스컬레이션

## MVP 범위 (2주 PoC)

**포함**
- FAQ 자동 응답 (상위 20개 질문)
- 주문 상태 조회 + 안내
- 간단한 배송 문의 처리

**제외 (2차)**
- 환불 자동 처리 (승인 흐름 필요)
- 복잡한 클레임 처리
- 다국어 지원
