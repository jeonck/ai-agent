멀티에이전트는 **진짜 필요할 때만** 도입하세요. 대부분의 문제는 단일 에이전트로 해결됩니다.

## 멀티에이전트가 필요한 상황

- 작업이 너무 길어 단일 컨텍스트 윈도우를 초과
- 독립적 검증이 필요 (서로 다른 에이전트가 교차 검토)
- 전문화된 역할이 병렬로 실행되어야 함

## 패턴 1: 오케스트레이터-워커

```
[오케스트레이터]
    ├──→ [워커 A: 문서 분석]
    ├──→ [워커 B: 데이터 수집]
    └──→ [워커 C: 보고서 생성]
         ↓
    [결과 통합]
```

**특징**
- 오케스트레이터가 전체 계획·분배·통합 담당
- 워커는 특정 태스크에 집중
- 워커들이 병렬 실행 가능

**언제 사용**: 대형 작업을 전문화된 하위 작업으로 분해할 때

## 패턴 2: 핸드오프 패턴

```
[에이전트 A] ──컨텍스트 전달──→ [에이전트 B] ──→ [에이전트 C]
```

**특징**
- 각 에이전트가 자신의 역할을 완료한 후 다음 에이전트로 넘김
- 컨텍스트(상태)가 체인을 따라 전달
- 순차 처리에 적합

**언제 사용**: 단계별 전문성이 다르고 순서가 중요할 때

## 패턴 3: 매니저 패턴

```
[사용자]
    ↕
[매니저 에이전트]  ← 모든 커뮤니케이션 중재
    ├──→ [전문가 A]
    └──→ [전문가 B]
```

**특징**
- 매니저가 사용자와의 유일한 접점
- 내부 에이전트 구조를 사용자에게 숨김
- 매니저가 라우팅·조정·응답 통합 담당

## 패턴 4: 그룹챗 패턴

```
[에이전트 A] ←─→ [에이전트 B]
      ↑                ↑
      └──── [에이전트 C] ──┘
```

**특징**
- 에이전트들이 서로 직접 통신
- 토론·협의 방식으로 결론 도출
- 복잡도가 높아 예측 어려움

{{< callout type="warning" >}}
**그룹챗 주의**: 에이전트 간 루프가 발생하기 쉽습니다. 항상 종료 조건과 최대 라운드 수를 설정하세요.
{{< /callout >}}

## 싱글에서 멀티로 넘어가는 기준

{{< callout type="info" >}}
아래 신호 중 **2개 이상**이 나타날 때 멀티에이전트를 고려하세요.
{{< /callout >}}

- 단일 에이전트의 컨텍스트가 지속적으로 한계에 도달
- 같은 작업을 여러 관점에서 독립 검증해야 함
- 진짜 병렬 전문 역할이 필요
- 단일 에이전트의 오류 복구가 일관되지 않음
