## 최적 패턴 선택 의사결정 트리

```
에이전트가 필요하다고 판단됨
         │
         ▼
사람 승인이 중간에 필요? ──YES──→ 승인 삽입형 에이전트
         │
        NO
         ▼
입력 유형이 다양하고 처리 방식이 달라야 함? ──YES──→ 라우팅
         │
        NO
         ▼
독립적 하위 작업들이 있음? ──YES──→ 병렬화
         │
        NO
         ▼
순차 단계가 필요? ──YES──→ 체이닝
         │
        NO
         ▼
반복 개선이 필요? ──YES──→ 평가자-최적화자
         │
        NO
         ▼
컨텍스트 한계 초과 / 전문 역할 분리? ──YES──→ 멀티에이전트
         │
        NO
         ▼
단일 루프형 에이전트
```

## 안티패턴: Over-agentification

{{< callout type="error" >}}
**가장 흔한 실수**: 단순한 문제에 복잡한 에이전트를 도입하는 것
{{< /callout >}}

**이런 경우 에이전트가 필요 없습니다**
- 단순 FAQ 응답 → RAG 검색 + 챗봇으로 충분
- 정해진 순서의 자동화 → 코드/RPA로 충분
- 단일 LLM 호출로 해결 → 에이전트 루프 불필요
- 실시간 응답 필요 → 에이전트의 여러 LLM 호출이 지연 유발
