Anthropic은 에이전트 성능의 핵심을 **"가장 작은 고신호 토큰 집합"**을 다루는 문제로 정의합니다.

{{< callout type="info" >}}
2026년 실무에서 에이전트 성능을 가장 크게 좌우하는 기술은 모델 크기가 아니라 **컨텍스트 엔지니어링**입니다.
{{< /callout >}}

## 컨텍스트 엔지니어링이란

에이전트에게 필요한 정보를 **적시에, 올바른 형태로, 최소한의 노이즈와 함께** 제공하는 기술

**나쁜 컨텍스트**
- 너무 많은 무관한 정보 → LLM 집중력 저하
- 오래된 정보 → 잘못된 판단
- 구조화되지 않은 정보 → 이해 오류

**좋은 컨텍스트**
- 현재 태스크에 직접 관련된 정보만
- 최신 상태를 반영
- 명확한 구조와 레이블

## 핵심 기법 6가지

### 1. 시스템 프롬프트 구조화

역할, 목표, 제약, 도구 사용 방법, 출력 형식을 **섹션으로 분리**합니다.

```xml
<role>당신은 고객지원 에이전트입니다.</role>
<goal>고객의 문의를 해결하거나, 해결 불가 시 에스컬레이션합니다.</goal>
<constraints>
- 환불은 7일 이내 구매만 처리
- PII 정보는 응답에 포함 금지
</constraints>
<tools>
- search_orders: 주문 내역 조회
- process_refund: 환불 처리 (승인 필요)
</tools>
```

### 2. Just-in-Time Context Loading

모든 정보를 처음부터 로드하지 말고, **필요한 시점에만** 관련 정보를 추가합니다.

```
❌ 나쁜 예: 시작 시 100페이지 매뉴얼 전체 로드
✅ 좋은 예: 고객이 환불 언급 시 환불 정책만 검색해서 추가
```

### 3. Context Compaction (압축)

긴 대화나 작업 이력을 **요약**하여 컨텍스트 크기를 줄입니다.

전략:
- 주기적으로 대화 이력을 요약하고 원본 교체
- 완료된 태스크는 결과만 남기고 과정 삭제
- 슬라이딩 윈도우: 최근 N턴만 유지

### 4. Agentic Memory / Note-taking

에이전트가 작업 중 **구조화된 노트**를 유지하도록 설계합니다.

```json
{
  "task_progress": "단계 3/5 완료",
  "findings": ["계약서에 위약금 조항 발견", "갱신일: 2026-09-01"],
  "pending_decisions": ["법무팀 검토 필요: 2.3조항"],
  "completed_actions": ["이메일 초안 작성"]
}
```

### 5. Few-shot 예시 설계

단순히 예시를 나열하는 것이 아니라, **경계 케이스와 실패 케이스**를 포함합니다.

```
✅ 좋은 예시 선택 기준:
- 전형적인 성공 케이스 (2-3개)
- 처리하기 까다로운 엣지 케이스 (1-2개)
- 거부해야 하는 케이스 (1-2개)
- 도구를 사용하는 케이스 (1-2개)
```

### 6. Sub-agent로 컨텍스트 분리

단일 에이전트의 컨텍스트가 너무 복잡해질 때, **서브에이전트로 분리**합니다.

```
오케스트레이터 (전체 계획, 작은 컨텍스트)
    ↓
서브에이전트 A (특정 태스크, 독립 컨텍스트)
서브에이전트 B (다른 태스크, 독립 컨텍스트)
```

## 컨텍스트 품질 체크리스트

- [ ] 시스템 프롬프트가 500 토큰 이내인가?
- [ ] Just-in-time 로딩을 적용했는가?
- [ ] 오래된 대화 이력을 압축하는가?
- [ ] 에이전트가 진행 상황을 노트로 기록하는가?
- [ ] Few-shot 예시가 경계 케이스를 포함하는가?
