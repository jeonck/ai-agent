에이전트는 세 가지 핵심 요소로 구성됩니다: **모델, 도구, 지침**

## 1. 모델 선택

모델 선택은 성능·비용·지연의 트레이드오프입니다.

| 기준 | 고려사항 |
|------|---------|
| **태스크 복잡도** | 단순 분류 → 소형 모델, 복잡 추론 → 대형 모델 |
| **지연 요구사항** | 실시간 응답 → 빠른 모델 우선 |
| **비용** | 입출력 토큰 단가 × 예상 볼륨 계산 |
| **컨텍스트 길이** | 긴 문서 처리 시 컨텍스트 윈도우 크기 확인 |
| **도구 사용 품질** | 모델별 도구 호출 정확도 차이 큼 |

**모델 선택 원칙**
- 오케스트레이터(계획·판단)에는 고성능 모델
- 특정 태스크 워커에는 비용 최적화 모델
- 라우팅 단계에는 소형 빠른 모델

## 2. 도구 설계

도구는 에이전트가 외부 세계와 상호작용하는 유일한 방법입니다.

**좋은 도구의 조건**
```
✅ 단일 책임 — 하나의 명확한 기능
✅ 명확한 명세 — LLM이 이해 가능한 description
✅ 안전한 실패 — 에러 메시지가 에이전트 이해 가능
✅ 멱등성 고려 — 중복 호출 시 부작용 최소화
✅ 최소 권한 — 필요한 권한만 부여
```

**도구 description 작성 원칙**
- LLM이 "언제 이 도구를 써야 하는지" 알 수 있도록 작성
- 파라미터 형식·범위·예시를 명시
- 반환값 구조 설명

## 3. 지침 (System Prompt) 설계

Anthropic 권장 시스템 프롬프트 구조:

```
1. 역할과 목표
   "당신은 [역할]입니다. [목표]를 달성합니다."

2. 행동 지침
   - 해야 하는 것
   - 하지 말아야 하는 것

3. 도구 사용 가이드
   - 각 도구를 언제 사용하는지
   - 도구 호출 순서 (있는 경우)

4. 출력 형식
   - 응답 형식 (JSON, 마크다운 등)
   - 길이·구조 요구사항

5. 예외 처리
   - 모르는 경우 어떻게 할 것인지
   - 실패 시 행동
```

## 4. 상태·세션 관리

에이전트가 장시간 실행될 때 상태를 어떻게 관리하는가:

| 상태 유형 | 설명 | 저장 방법 |
|---------|------|---------|
| **단기 메모리** | 현재 대화/태스크 컨텍스트 | 컨텍스트 윈도우 |
| **작업 노트** | 진행 중 중간 결과 | 구조화된 노트 (도구) |
| **장기 메모리** | 세션 간 유지 정보 | 외부 DB/벡터 저장소 |
| **에피소딕 메모리** | 과거 상호작용 패턴 | 벡터 DB |

{{< callout type="info" >}}
컨텍스트가 길어질수록 LLM 성능이 저하됩니다. 상태를 압축하거나 외부에 오프로드하는 전략이 필요합니다.
{{< /callout >}}
