매일 쏟아지는 업무용 이메일, 그 안에서 중요한 마감 기한과 요청 사항을 일일이 찾아 정리하느라 시간을 낭비하고 계신가요?

오늘은 단순히 대화만 주고받는 챗봇을 넘어, **실제 업무 프로세스를 자동으로 수행하는 'AI 에이전트'** 를 구축하는 방법을 공유합니다. 복잡한 코딩 없이, 5가지 파일 구성만으로 업무 효율을 크게 높일 수 있습니다.

---

## 챗봇과 에이전트, 뭐가 다를까요?

먼저 둘의 차이를 직관적으로 비교해봅시다.

| 구분 | 챗봇 | AI 에이전트 |
|------|------|------------|
| 역할 | 질문에 답변 | 목표를 달성하기 위해 행동 |
| 연결 | 없음 | Gmail, 캘린더, 슬랙 등 실제 툴 |
| 예시 | "이메일 요약해줘" → 텍스트 반환 | Gmail 직접 열람 → 우선순위 분석 → 할 일 목록 자동 생성 |

쉽게 말해, 챗봇은 **말만 하는 조수**이고, 에이전트는 **직접 일을 처리하는 비서**입니다.

---

## 실전 예제: 이메일 비서 에이전트

구체적인 예시로 이해해봅시다.

**시나리오**: 매일 아침 "할 일 추출해줘"라는 한 마디만 입력하면, AI가 오늘 받은 이메일 전체를 분석해 마감 기한순으로 정리된 할 일 목록을 만들어 주는 에이전트를 구축합니다.

**결과물 예시**:
[긴급] A사 계약서 검토 회신 (마감: 오늘 오후 3시) - 보낸 사람: kim@a-company.com

[중요] B 프로젝트 주간 보고서 제출 (마감: 내일 오전 10시) - 보낸 사람: 팀장

[보통] 월간 뉴스레터 구독 확인 요청 - 무시 가능


이 에이전트를 만드는 데 필요한 것은 딱 **5개의 파일**입니다.

---

## 5가지 핵심 파일 구성

### 파일 1. `01-start-here.html` — 정체성 정의

> "당신은 누구이고, 언제 깨어나는가"

에이전트가 어떤 역할을 맡는지, 어떤 트리거 문구에 반응하는지를 정의합니다.

```html
<agent-identity>
  역할: 나의 이메일 관리 비서
  이름: 메일봇
  트리거 문구: "할 일 추출해줘"
  
  역할 설명:
  사용자의 Gmail을 열람하여 오늘 받은 이메일 중
  행동이 필요한 항목을 추출하고, 우선순위별로 정리한다.
</agent-identity>
```

**핵심 포인트**: 트리거 문구를 명확하게 설정할수록, 에이전트가 언제 개입해야 하는지 정확히 인식합니다.

---

### 파일 2. `02-golden-example.html` — 결과물의 표준

> "내가 원하는 완벽한 출력이 바로 이것이다"

AI에게 '이상적인 답변'의 예시를 직접 보여줍니다. 이것이 곧 품질의 기준점이 됩니다.

```html
<golden-example>
  입력: "할 일 추출해줘"
  
  이상적인 출력:
  ---
  📋 오늘의 할 일 목록 (2026-06-20 기준)
  
  🔴 긴급 (오늘 처리 필요)
  1. [A사] 계약서 검토 후 회신
     마감: 오늘 오후 3시 | 발신: kim@a-company.com
  
  🟡 중요 (내일까지)
  2. [팀 내부] B 프로젝트 주간 보고서 제출
     마감: 내일 오전 10시 | 발신: 팀장(lee@mycompany.com)
  
  ⚪ 보통 (이번 주 내)
  3. 콘퍼런스 참석 여부 회신
     마감: 이번 주 금요일 | 발신: event@conference.com
  ---
</golden-example>
```

**핵심 포인트**: 예시가 구체적일수록 실제 출력 품질이 높아집니다. 포맷, 이모지, 순서까지 원하는 대로 보여주세요.

---

### 파일 3. `03-process.html` — 사고의 로직

> "어떤 순서로 생각하고 행동하는가"

AI가 데이터를 분석하는 단계별 절차를 명시합니다. 이 파일이 에이전트의 '두뇌 회로'입니다.

```html
<process>
  Step 1. Gmail에서 오늘 받은 이메일 전체 목록 가져오기
  
  Step 2. 각 이메일을 분석하여 아래 기준으로 분류:
    - 행동 필요 여부 (답장, 검토, 제출 등)
    - 마감 기한 존재 여부
    - 발신자 중요도 (04-context.html 참조)
  
  Step 3. 우선순위 결정:
    - 긴급: 오늘 내 마감 또는 발신자가 VIP
    - 중요: 내일~이번 주 마감
    - 보통: 마감 불분명 또는 낮은 우선순위
  
  Step 4. 02-golden-example.html 형식에 맞게 목록 출력
  
  Step 5. 05-quality.html 체크리스트 통과 후 최종 전달
</process>
```

**핵심 포인트**: 단계를 명확히 쪼갤수록 AI가 논리적으로 판단합니다. "알아서 해줘"보다 "이 순서로 해줘"가 훨씬 정확합니다.

---

### 파일 4. `04-context.html` — 개인화 정보

> "나는 누구이고, 내 업무 환경은 어떻게 생겼나"

이 파일이 에이전트를 '범용 비서'가 아닌 '나만의 비서'로 만드는 핵심입니다.

```html
<context>
  사용자 정보:
  - 이름: 김철수
  - 직책: 마케팅 팀장
  - 회사: 스타트업 A (직원 30명)

  VIP 발신자 (무조건 긴급 처리):
  - 대표이사: ceo@mycompany.com
  - 주요 클라이언트: kim@a-company.com, park@b-corp.com

  무시해도 되는 발신자:
  - 마케팅 수신 동의 뉴스레터
  - 자동 발송 알림 (no-reply@...)

  자주 쓰는 마감 표현:
  - "COB" = 오늘 업무 종료 전 (오후 6시)
  - "EOW" = 이번 주 금요일
</context>
```

**핵심 포인트**: 처음에는 간단하게 시작하고, 사용하면서 조금씩 추가하면 됩니다. 쓸수록 나에게 딱 맞는 비서가 됩니다.

---

### 파일 5. `05-quality.html` — 품질 검사

> "출력 전 스스로 한 번 더 확인한다"

AI가 최종 답변을 내놓기 전에 스스로 점검하는 체크리스트입니다. 오류와 환각(Hallucination)을 방지합니다.

```html
<quality-checklist>
  출력 전 아래 항목을 모두 확인하라:
  
  [ ] 모든 항목에 발신자와 마감 기한이 명시되어 있는가?
  [ ] 마감 기한이 없는 항목은 "보통"으로 분류했는가?
  [ ] VIP 발신자 이메일은 자동으로 "긴급"으로 처리했는가?
  [ ] 무시 대상 발신자 이메일은 목록에서 제외했는가?
  [ ] 출력 형식이 02-golden-example.html과 동일한가?
  [ ] 날짜와 시간 정보가 정확한가? (임의로 추측하지 않았는가?)
  
  위 항목 중 하나라도 미충족 시, 해당 부분을 수정 후 재출력한다.
</quality-checklist>
```

## 구축부터 실행까지: 3단계 흐름
[1단계] 파일 준비          [2단계] 연결              [3단계] 실행

─────────────────         ────────────────         ────────────────

5개 파일 작성     →   에이전트 플랫폼에    →   트리거 문구 입력

(구글 드라이브       지식 베이스로 등록        "할 일 추출해줘"

폴더에 저장)        + Gmail 연결                      ↓

AI가 5개 파일 조합

→ 맞춤형 할 일 목록 출력
**1단계**: 구글 드라이브에 `나의-AI-비서` 폴더를 만들고, 위 5개 파일을 작성해 저장합니다.

**2단계**: Claude Cowork 등 에이전트 지원 플랫폼에서 해당 폴더를 Knowledge Base로 등록하고, Gmail 연동을 추가합니다.

**3단계**: "할 일 추출해줘" 한 마디만 입력하면 끝입니다.

## 더 나아가기: 응용 아이디어

같은 구조로 아래와 같은 에이전트도 만들 수 있습니다.

- **회의록 비서**: 회의 녹취 파일 → 핵심 결정 사항 + 담당자별 액션 아이템 자동 정리
- **리서치 비서**: 키워드 입력 → 관련 뉴스 수집 + 요약 보고서 생성
- **고객 응대 비서**: 문의 이메일 → 유형 분류 + 답변 초안 자동 작성

5개 파일의 내용만 바꾸면 전혀 다른 에이전트가 탄생합니다.
